Revamping
SpeedQueen

A full-cycle redesign of the SpeedQueen laundry app used by Rutgers students — from user interviews and affinity mapping to hi-fi prototyping and usability testing.

01 — Overview

The App &
The Problem

SpeedQueen is a mobile app that lets Rutgers on-campus students pay for and monitor laundry machines in their dorms. While convenient for going coin-free, the existing app had significant usability issues that frustrated everyday users.

Our team identified four key problem areas that drove the need for a redesign.

Navigation

Confusing machine selection flow and unclear setup for first-time users.

Transactions

No refund option, no custom reload, and a $10 minimum load requirement.

Feedback

No push notifications when laundry finishes — users relied on personal timers.

Personalization

Balances locked to one campus, no machine favorites, minimal customization.

03 — Empathize

Listening to
Real Users

We wrote a 30-question semi-structured interview script organized around five themes, then conducted 30-minute recorded sessions with Rutgers students. Recordings were transcribed using TurboScribe and pulled into raw quote data.

04 — Define

Synthesizing
the Data

Interview notes were clustered into an affinity diagram, surfacing six major themes: lack of communication, user motivations, slider issues, transaction limitations, cached memory problems, and interaction habits.

01

Communication Issues

Lack of descriptions and visual feedback left users confused during onboarding and after making payments.

02

Poor UI Elements

The time-selection slider was imprecise and frustrating; buttons were too small for confident interaction.

03

Transaction Limitations

No refund option, no custom reload amount, and the $10 minimum actively discouraged regular use.

05 — Ideate

Personas, Storyboards
& Solutions

We created user personas — including Terry, an 18-year-old freshman on a tight budget — to ground our thinking in real goals and frustrations. Storyboard sketches mapped the full user journey and surfaced moments where redesign could reduce friction.

Replacing the Slider

Swapped the janky slider for distinct time-increment buttons that highlight on selection — clear, tappable, and mistake-proof.

Adding a Refund Flow

Designed a full refund request flow: choose amount → select payout method → confirm → receive confirmation with updated balance.

07 — Test

Usability Testing
Findings

We ran in-person usability testing sessions with participants navigating the hi-fi prototype on a phone. Three categories of issues surfaced.

Navigation — Lack of Scrolling

The prototype didn't support scroll gestures, causing users to miss below-the-fold content.

Labeling — Counterintuitive Patterns

Some users searched the sidebar for location options; others tapped the balance expecting it to open the wallet.

Branding — Location Context

After selecting Rutgers, users expected Rutgers-specific branding — a missed opportunity to build contextual trust.

Next Steps

Enable scrolling and responsive gestures; make the balance a tappable shortcut to the wallet; introduce location-aware branding post-login.

08 — Reflection

What We
Learned

Key Takeaways

  • Interviews surfaced pain points no audit would have caught
  • Affinity diagramming revealed patterns across unrelated complaints
  • Small UI changes have outsized impact on user trust
  • Designs must match users' existing mental models

If We Had More Time

  • Apple Pay / Google Pay integration
  • Machine favorites and usage history
  • Cross-campus balance transfer
  • Full accessibility audit
  • Second round of usability testing
